# Testing a Feature Locally

This guide covers the two main ways to validate features before pushing: the full local CI gate and targeted Dev Container CLI commands.

## Run the full local CI gate

`scripts/local-ci.sh` is a convenience script that runs the same checks the repository CI runs, but on your machine. It is not invoked by GitHub Actions; it exists to catch failures early.

### Prerequisites

- Docker installed and running.
- [act](https://github.com/nektos/act) installed.
- [gh](https://cli.github.com/) CLI authenticated.
- [uv](https://docs.astral.sh/uv/) installed.

### Run it

From the repository root:

```bash
./scripts/local-ci.sh
```

The script performs the following checks and prints `PASS`/`FAIL` for each:

1. **Prerequisites** — `act`, `docker`, and a running daemon.
2. **shellcheck** — on `install.sh`, `uninstall.sh`, test scripts, and helper scripts.
3. **README sync** — `uv run python scripts/generate-feature-readmes.py --check`.
4. **Workflow validation** — `act -j validate` runs `.github/workflows/validate.yml`.
5. **Dry-run release** — `act -j deploy --dryrun` exercises `.github/workflows/release.yaml` without publishing.
6. **Feature smoke tests** — default-install tests for every feature via `npx @devcontainers/cli features test`.

Fix any failures before pushing. The gate exits with code `0` only when every check passes.

## Test a single feature with the Dev Container CLI

For faster feedback during development, run the CLI directly against one feature.

### Install the CLI

```bash
npm install -g @devcontainers/cli
```

VS Code also ships a bundled binary:

```bash
~/.config/Code/User/globalStorage/ms-vscode-remote.remote-containers/cli-bin/devcontainer
```

### Default-install smoke test

Build a container with only the default options for a feature:

```bash
devcontainer features test \
    --skip-scenarios \
    -f claude-code-hooks \
    -i mcr.microsoft.com/devcontainers/base:ubuntu \
    .
```

- `--skip-scenarios` — skip the `scenarios.json` matrix; use this when iterating on the install script.
- `-f claude-code-hooks` — target one feature.
- `-i <image>` — base image for the test container.

### Scenario tests

Run only the per-feature scenarios defined in `test/<feature>/scenarios.json`:

```bash
devcontainer features test \
    -f claude-code-hooks \
    --skip-autogenerated \
    .
```

### Global integration scenarios

Run cross-feature scenarios from `test/_global/scenarios.json`:

```bash
devcontainer features test --global-scenarios-only .
```

## Interpret smoke test results

The CLI returns `0` when all tests pass and a non-zero exit code when something fails. During the run you will see:

- The container image being built.
- Feature `install.sh` output.
- The test script output.
- A summary line for each scenario.

Common failure modes:

- **Build failure in `install.sh`** — a command exited non-zero. Fix the script and rerun.
- **Missing option values** — defaults are not what you expected; check `devcontainer-feature.json`.
- **Network dependency fails** — some features clone external repos. Add `skipOnFailure` or mock the dependency for local tests.
- **Stale cache** — the CLI may reuse a cached feature layer. Clean caches or pass `--build-no-cache`.

## Clean caches when tests do not pick up changes

If a change to a feature does not appear in tests, remove stale caches:

```bash
# Remove devcontainer CLI feature cache
rm -rf /tmp/devcontainercli-*/container-features/*

# Remove generated containers and images
docker ps -aq --filter label=devcontainer.local_folder | xargs -r docker rm -f
docker images --format "{{.Repository}}:{{.Tag}}" | grep "vsc-" | xargs -r docker rmi -f

# Remove a lockfile that may pin an older version
rm -f .devcontainer/devcontainer-lock.json
```
