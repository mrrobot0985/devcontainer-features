# CI Workflows

All CI runs on every push to `main` and on every pull request. Workflows are split so each check has a single responsibility. The checks listed below are required to pass before a PR can merge.

Release publishing is separate: it runs only when a developer pushes a prefixed tag (or manually dispatches the release workflow for backfill).

## Scope of this repository

This monorepo only ships **differentiators**: Claude Code policy/hooks/skills, security audit and enforcement, devcontainer meta tooling, and unique stacks. Pure “install CLI X” tools live in [official](https://containers.dev/features) or community feature registries — not here. That keeps the tree small and CI honest.

## Workflow summary

| Workflow                                     | File                                         | Trigger                       | What it checks / does                                                                                                       |
| -------------------------------------------- | -------------------------------------------- | ----------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **CI - Test Features**                       | `.github/workflows/test.yaml`                | push to `main`, PR, manual    | Builds selected features in autogenerated default tests, per-feature scenario tests, and global integration scenarios.      |
| **Validate devcontainer-feature.json files** | `.github/workflows/validate.yml`             | push to `main`, PR, manual    | Validates JSON metadata schema, runs shellcheck on scripts, and checks that feature READMEs are in sync with JSON metadata. |
| **Lint workflows**                           | `.github/workflows/lint-workflows.yml`       | push to `main`, PR, manual    | Lints all GitHub Actions workflow files with `actionlint`.                                                                  |
| **Conventional Commits**                     | `.github/workflows/conventional-commits.yml` | PR open/edit/sync             | Ensures the PR title follows Conventional Commits.                                                                          |
| **Release dev container features**           | `.github/workflows/release.yaml`             | push of `*-v*` tag, or manual | Publishes features under `./src` to GitHub Container Registry. Does not create tags.                                        |

## CI - Test Features

File: `.github/workflows/test.yaml`

- `setup` — selects features:
  - **Pull request:** only features with paths changed under `src/<id>/` or `test/<id>/` (full set if `test.yaml` or `scripts/` change).
  - **main / manual:** every feature under `src/`.
- `test-autogenerated` — default-install on one base image (`mcr.microsoft.com/devcontainers/base:ubuntu`). Network-heavy features may be matrix-excluded and covered by scenarios.
- `test-scenarios` — selected features that have `test/<feature>/scenarios.json`.
- `test-global` — `test/_global/scenarios.json` (always one job).
- `test-summary` — fails if any required job failed; empty feature matrices are OK (e.g. docs-only PRs).

## Validate devcontainer-feature.json files

File: `.github/workflows/validate.yml`

- `validate` — schema validation via `devcontainers/action@v1`
- `shellcheck` — installers, uninstallers, tests, helpers
- `readme-sync` — `generate-feature-readmes.py --check`
- `docs-check` — `mdformat --check`

## Lint workflows

File: `.github/workflows/lint-workflows.yml`

Runs `actionlint` on every workflow file.

## Conventional Commits

File: `.github/workflows/conventional-commits.yml`

Validates PR titles: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `ci`, `build`, `perf`.

## Release dev container features

File: `.github/workflows/release.yaml`

Triggered by developer-pushed `*-v*` tags or manual `workflow_dispatch`. Does not create tags.

## Required status checks

- `test.yaml`
- `validate.yml`
- `lint-workflows.yml`
- `conventional-commits.yml`
