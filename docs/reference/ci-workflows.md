# CI Workflows

All CI runs on every push to `main` and on every pull request. Workflows are split so each check has a single responsibility. The checks listed below are required to pass before a PR can merge.

## Workflow summary

| Workflow                                     | File                                         | Trigger                    | What it checks                                                                                                                                          |
| -------------------------------------------- | -------------------------------------------- | -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **CI - Test Features**                       | `.github/workflows/test.yaml`                | push to `main`, PR, manual | Builds each feature in autogenerated default tests, per-feature scenario tests, and global integration scenarios. Fails if any required test job fails. |
| **Validate devcontainer-feature.json files** | `.github/workflows/validate.yml`             | push to `main`, PR, manual | Validates JSON metadata schema, runs shellcheck on scripts, and checks that feature READMEs are in sync with JSON metadata.                             |
| **Lint workflows**                           | `.github/workflows/lint-workflows.yml`       | push to `main`, PR, manual | Lints all GitHub Actions workflow files with `actionlint`.                                                                                              |
| **Conventional Commits**                     | `.github/workflows/conventional-commits.yml` | PR open/edit/sync          | Ensures the PR title follows Conventional Commits.                                                                                                      |
| **Auto-release changed features**            | `.github/workflows/auto-release.yml`         | weekly schedule, manual    | Bumps the patch version for any feature whose source directory changed since its latest prefixed tag, then opens a release PR.                          |
| **Tag feature releases**                     | `.github/workflows/tag-release.yml`          | push to `main`, manual     | Reads each feature's current version from JSON and creates the prefixed git tag if it does not already exist.                                           |
| **Release dev container features**           | `.github/workflows/release.yaml`             | push of `*-v*` tag         | Publishes the affected feature to GitHub Container Registry.                                                                                            |

## CI - Test Features

File: `.github/workflows/test.yaml`

This is the primary test gate. It has four jobs:

- `setup` — emits the list of features so downstream matrices can consume it.
- `test-autogenerated` — default-install tests across a matrix of base images (`debian:latest`, `ubuntu:latest`, `mcr.microsoft.com/devcontainers/base:ubuntu`). `claude-code-skills` and `claude-code-plugins` are excluded because they perform network-dependent clones/installs by default.
- `test-scenarios` — per-feature scenario tests from `test/<feature>/scenarios.json`.
- `test-global` — cross-feature integration scenarios from `test/_global/scenarios.json`.
- `test-summary` — runs even when earlier jobs fail and fails the workflow if any required test job failed.

## Validate devcontainer-feature.json files

File: `.github/workflows/validate.yml`

Three parallel jobs:

- `validate` — uses `devcontainers/action@v1` with `validate-only: "true"` to validate every `devcontainer-feature.json` against the devcontainer features schema.
- `shellcheck` — runs `shellcheck --severity=warning` on `scripts/local-ci.sh`, `.githooks/pre-commit`, `src/*/install.sh`, `src/*/uninstall.sh`, and all test scripts.
- `readme-sync` — runs `uv run python scripts/generate-feature-readmes.py --check` to ensure each feature README matches its JSON metadata.

## Lint workflows

File: `.github/workflows/lint-workflows.yml`

Installs `actionlint` and runs it against every workflow file in `.github/workflows/`. Catches syntax errors, missing permissions, insecure expressions, and Actions-specific issues.

## Conventional Commits

File: `.github/workflows/conventional-commits.yml`

Runs `amannn/action-semantic-pull-request@v5` on PR open, edit, and synchronize events. Validates the PR title against the Conventional Commits types used in this repository:

`feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `ci`, `build`, `perf`.

Because squash merges use the PR title as the commit message, this keeps `main` branch history consistent.

## Auto-release changed features

File: `.github/workflows/auto-release.yml`

Runs every Sunday at 03:17 UTC and on demand. For each feature with a source directory that changed since its latest prefixed tag, it bumps the patch version in `devcontainer-feature.json`. When any feature changed, it opens a PR titled `chore: bump feature versions` from branch `release/bump-feature-versions`.

## Tag feature releases

File: `.github/workflows/tag-release.yml`

Runs on every push to `main` and on demand. For each feature, it derives the expected tag from `devcontainer-feature.json` (`<feature-id>-v<version>`) and creates the annotated tag through the GitHub API when it does not already exist.

## Release dev container features

File: `.github/workflows/release.yaml`

Triggered only by prefixed tags (`*-v*`). Publishes the feature whose source directory matches the tag prefix to GitHub Container Registry using `devcontainers/action@v1` with `publish-features: "true"` and `base-path-to-features: "./src"`.

## Required status checks

The following workflows are required to pass before a PR merges:

- `test.yaml`
- `validate.yml`
- `lint-workflows.yml`
- `conventional-commits.yml`
