# CI Workflows

All CI runs on every push to `main` and on every pull request. Workflows are split so each check has a single responsibility. The checks listed below are required to pass before a PR can merge.

Release publishing is separate: it runs only when a developer pushes a prefixed tag (or manually dispatches the release workflow for backfill).

## Scope of this repository

This monorepo only ships **differentiators**: Claude Code policy/hooks/skills, security audit and enforcement, devcontainer meta tooling, and unique stacks. ## CI - Test Features

File: `.github/workflows/test.yaml`

- `setup` â€” selects features:
  - **Pull request:** only features with paths changed under `src/<id>/` or `test/<id>/` (full set if `test.yaml` or `scripts/` change).
  - **main / manual:** every feature under `src/`.
- `test-autogenerated` â€” default-install on one base image (`mcr.microsoft.com/devcontainers/base:ubuntu`). Network-heavy features may be matrix-excluded and covered by scenarios.
- `test-scenarios` â€” selected features that have `test/<feature>/scenarios.json`.
- `test-global` â€” `test/_global/scenarios.json` (always one job).
- `test-summary` â€” fails if any required job failed; empty feature matrices are OK (e.g. docs-only PRs).

## Validate devcontainer-feature.json files

File: `.github/workflows/validate.yml`

- `validate` â€” schema validation via `devcontainers/action@v1`
- `shellcheck` â€” installers, uninstallers, tests, helpers
- `readme-sync` â€” `generate-feature-readmes.py --check`
- `docs-check` â€” `mdformat --check`

## Lint workflows

File: `.github/workflows/lint-workflows.yml`

Runs `actionlint` on every workflow file.

## Conventional Commits

File: `.github/workflows/conventional-commits.yml`

Validates PR titles: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `ci`, `build`, `perf`.

## Release dev container features

File: `.github/workflows/release.yaml`

Triggered by developer-pushed `*-v*` tags or manual `workflow_dispatch`. Does not create tags.

Before publish, job `ghcr-lag-gate` runs `scripts/check-ghcr-majors.sh` (fail closed). The gate does **not** re-publish packages.

## GHCR major lag gate

File: `.github/workflows/ghcr-lag-gate.yml`

Daily schedule plus path-filtered push/PR. Verifies major floating tags for the critical template-consumed set (and templates/src when present in a monorepo checkout). See [Scripts and Automation](scripts.md).
