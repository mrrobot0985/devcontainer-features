# Multi-Perspective Review System

## Purpose

Replace the single-thread `validate-branch.sh` gate with three isolated review perspectives that run in parallel inside Docker sandcastles. Each perspective examines the same diff/branch from a different angle. Results are aggregated into a single JSON report that controls the CI gate.

## Perspectives

### 1. Correctness

**Question:** Does the code work, parse, and pass its own test suite?

**Checks:**

- Shell scripts parse (`bash -n`, `shellcheck`)
- Python/JS/TS files parse and type-check (project-dependent; detected from repo contents)
- Unit/integration tests pass (`pytest`, `npm test`, `go test`, etc.)
- No unresolved merge conflicts or syntax errors

**Failure modes:** non-zero exit from any tool, test failures, parse errors.

**Severity mapping:**

- `critical`: syntax/parse error, test failure
- `warning`: shellcheck warning, type-check warning
- `info`: optional coverage or timing notes

### 2. Architecture

**Question:** Does the change fit existing patterns and avoid structural rot?

**Checks:**

- Directory/file layout matches `CONTRIBUTING.md` / `CODING_STANDARDS.md`
- No import/require cycles (language-specific tooling)
- No obvious duplication across changed files
- No speculative abstractions (parameters, hooks, or base classes not justified by the diff)
- No middle-man modules or refused bequests in the changed code
- Public API surface changes are intentional and documented

**Tools:** `pydeps`, `jscpd`, simple AST/grep heuristics, plus a documented pattern file for the repo.

**Severity mapping:**

- `critical`: circular dependency, public API breakage without docs
- `warning`: smell hit (duplication, speculative generality, etc.)
- `info`: structural observations

### 3. Safety

**Question:** Does the change respect safety, policy, and repo invariants?

**Checks:**

- Secret scanning (`gitleaks`, `truffleHog`, or `git-secrets`)
- No AI attribution in commits or diff (`Generated by`, `Co-Authored-By:`, etc.)
- Conventional commits (`commitlint`)
- No destructive operations without explicit human-approval markers
- No inline git config overrides
- No private keys, `.env` files, or credential patterns committed

**Severity mapping:**

- `critical`: detected secret, attribution in commit, non-conventional commit on protected branch
- `warning`: destructive operation present but gated by approval marker, suspicious pattern
- `info`: policy reminders

## Orchestration

```
                    review-orchestrator.sh
                            |
        +-------------------+-------------------+
        |                   |                   |
  correctness           architecture          safety
  (container)           (container)         (container)
        |                   |                   |
        +-------------------+-------------------+
                            |
                    aggregate-reports.js
                            |
                    JSON report + exit code
```

### Flow

1. Orchestrator receives target ref (branch or PR merge ref) and optional base ref.
2. Orchestrator clones the repo into a temporary, read-only workspace.
3. Three containers start in parallel from a shared base image:
   - `/usr/local/bin/review-correctness.sh`
   - `/usr/local/bin/review-architecture.sh`
   - `/usr/local/bin/review-safety.sh`
4. Each container writes its own JSON report to a shared output volume.
5. Orchestrator waits for all containers, then runs aggregation.
6. Aggregator computes overall verdict and emits `report.json`.
7. Exit code is `0` only when all perspectives pass with zero critical findings.

### Concurrency

- Use `docker run --detach` for each perspective and poll/wait via `docker wait`.
- Timeout per perspective: 10 minutes (configurable).
- Shared volume mounted at `/out` for reports; repo mounted read-only at `/repo`.

## Reporting Format

Each perspective emits a JSON fragment matching `schemas/review-report.json`. The aggregator combines them into a single report.

```json
{
  "run_id": "uuid",
  "target_ref": "feat/multi-perspective-review",
  "base_ref": "main",
  "timestamp": "2026-07-15T12:00:00Z",
  "overall": "pass|fail",
  "perspectives": [
    {
      "name": "correctness",
      "status": "pass",
      "exit_code": 0,
      "duration_ms": 42000,
      "findings": []
    }
  ],
  "summary": {
    "critical": 0,
    "warning": 0,
    "info": 0
  }
}
```

### Finding object

```json
{
  "perspective": "safety",
  "severity": "critical",
  "rule": "no-secrets",
  "file": "src/foo/install.sh",
  "line": 17,
  "message": "possible API key pattern matched",
  "suggestion": "rotate the credential and purge from history"
}
```

## Gating Logic

1. If any perspective exits non-zero → `overall: fail`.
2. If any critical finding exists → `overall: fail`.
3. If warnings exceed `max_warnings` (default 10) → `overall: fail`.
4. Otherwise → `overall: pass`.

CI/ralph-loop consumes the report:

- `overall == pass`: approve and continue (merge if policy allows).
- `overall == fail`: block merge, surface the report, and stop the current ralph loop cycle.

## Docker Sandcastle Layout

```
sandcastle/
  Dockerfile.base           # shared Ubuntu base with jq, git, curl
  Dockerfile.correctness    # adds shellcheck, python, pytest, npm
  Dockerfile.architecture   # adds pydeps, jscpd, pattern linter
  Dockerfile.safety         # adds commitlint, gitleaks, policy regex
```

Each container runs with:

- `--read-only` root filesystem
- repo mounted read-only
- `/out` as a writable volume for that perspective only
- no network unless required (Correctness may need package installs; Safety runs offline)

## Implementation Approach

1. Create the report JSON schema first so both perspective scripts and the aggregator share a contract.
2. Build the three perspective shell scripts; keep them deterministic and free of LLM improvisation.
3. Build a single `review-orchestrator.sh` that coordinates the Docker containers and aggregates output.
4. Add a GitHub Actions workflow that invokes the orchestrator on every PR and push.
5. Wire the orchestrator into `ralph-loop.sh` so each implementation cycle ends with a multi-perspective review before merge.
6. Iterate the safety ruleset from the repo's `.claude/rules/` invariants (no attribution, no secrets, conventional commits, human sovereignty, no git config override).
